ELK日志平台（二）：Nxlog初探

发表于 2015-08-31   |  

　　Nxlog是一款日志管理程序，实现日志的收集（shipper）功能，可以在Windows和Linux下运行。有社区版（Community Edition）和商业版（Enterprise Edition）两个版本。我当然是使用社区版啦 ; )
　　选用Nxlog的原因是公司有一些Windows平台上的日志需要收集，而Rsyslog实在是难用（包括在Linux上）。不用自己造轮子的情况下Nxlog几乎是唯一的选择。使用下来感觉非常不错，比Rsyslog好用多了，哼~
　　
　　Ok,Let’s begin.

安装配置

安装环境


　　- os：Windows Server 2008 R2 Enterprise x64
　　- pkg：nxlog-ce-2.9.1347.msi

安装步骤

  1. 下载之后一路next即可。
  2. 安装完成后nxlog的根目录为C:\Program Files (x86)\nxlog。
  3. 同时在Windows的服务（services.msc）里也会多出来一个名为nxlog的服务，此时服务是未开启的状态。

配置说明


　　nxlog的配置文件名为nxlog.conf，在目录下的.\conf下，默认为C:\Program Files (x86)\nxlog\conf\nxlog.conf。打开文件，内容如下：

全局变量

  • Moduledir 模块根目录
  • CacheDir 缓存根目录
  • Pidfile Pid文件路径
  • SpoolDir 工程文件目录
  • LogFile 日志文件路径

还可以设置的全局变量：

  • User 启动进程的用户（Windows下默认nxlog-processor）
  • Group 启动进程的用户组（Windows下默认nxlog-processor）
  • LogLevel 日志等级（CRITICAL, ERROR,WARNING, INFO, DEBUG，默认INFO）
  • FlowControl 流量控制

完整全局变量请在官方文档中查询使用。

Extension Module


nxlog和logstash一样，是采取模块化的设计，需要什么模块扩展加载即可

　　常用模块有：

  • CSV（xm_csv） 处理CSV格式文件
  • JSON（xm_json） 处理JSON格式字段
  • XML（xm_xml） 处理XML格式文档
  • Multi-line message parser (xm_multiline) 处理多行日志文件

所有扩展模块请在官方文档中查询使用。

Input Output工作流


接下来我们来梳理一下，日志收集整个流程：

Input(xm_xxx) ——> Porcessor(pm_xxx) ——> Output(om_xxx)

  1. Input，解决日志从哪里来的问题，通过扩展模块的方式，决定日志从File中读取（im_file），还是从TCP（im_tcp）或者UDP（im_udp）传输进来？默认配置的im_msvistalog，表示日志从im_msvistalog模块输入进来。至于im_msvistalog模块内部如何运作，输出日志的格式又是怎样，请到官方文档中寻找答案。
  2. Porcessor，解决了接收到日志之后，我要对它做什么的问题。是切割、过滤，还是添加字段、封装成JSON？
  3. Output，日志输入和处理完毕之后，要发送到哪里？可以直接写入文件（om_file），也可以通过TCP（om_tcp）传输到远端。甚至可以直接插入elasticsearch（om_elasticsearch，nxlog社区版并没有这个模块哈哈哈）建立索引？
  4. Exec，在上边配置output项目中，有一个Exec to_syslog_snare(); 这表示在ouput的时候，调用to_syslog_snare()方法。Exec可以在input、porcessor、output中调用，只是后边的方法一定要提前调用对应的Extension Module。
  5. Route，即路由。nxlog支持多个input、porcessor、output方式同时运行。input和output如何对应，则通过Route来实现。

后记


　　简单理解日志收集的工作流程，才能实现我们的心中所想，收集哪些日志，怎么过滤，怎么添加、删除字段，怎么保存等内容。下一篇博客将用一个生产环境中的例子来完整展示上述内容。

ELK日志平台（一）：框架结构

发表于 2015-08-21   |  

　　最近在公司搭建日志平台，使用的是集收集、存储、展示功能于一身的ELK三法宝套装：Logstash、Elasticsearch、Kibana！
　　在搞这个系统的时候，遇到了一些坑，也挠破了一些头皮。把具体的实现过程记录在这里，希望看到的同学能够少走一些弯路，赶紧搭建完毕升职加薪走上人生巅峰。也恳求老鸟不屑一顾随手指点批评指正。

总体架构：

　　网上找了一张图，整个日志收集平台大概就是这么个意思，整个架构由以下角色组成：

  • Shipper：作用是在应用服务器上收集日志，现在比较流行的工具有Flume、Logstash、Syslog等。
  • Broker：上图四个Redis Node组成的Cluster就是一个Broker，它的作用类似于一个队列，缓存由Shipper发送过来的日志条目。除了Redis，也有用Kafka来做的。
  • Indexer：中间三个Logstash，作用是从前面的Broker中取出日志条目，根据写好的规则匹配，转发到对应的Elasticsearch里。

有朋友问了，为什么Shipper和Indexer里都有Logstash？没错，这就是Logstash的强大之处。作为Shipper，它可以安装在应用服务器上作为日志收集的客户端。作为Indexer，它可以接收收到的日志，进行格式化后存入Elasticsearch并生成索引（这里的索引和关系型数据库索引的概念完全不一样）。因为Logstash需要JAVA运行环境，所以不推荐使用它来做Shipper，毕竟在每一台应用服务器上部署JAVA运行环境是不可接受的。

  • Elasticsearch：一个倒序索引的文档型数据库，用来存储收集的日志。这里的索引可以理解成关系型数据库的一个database（下回会深入地聊一聊这个索引）。Elasticsearch还提供了Restful风格的Web API，可以方便地对数据进行增删改查等操作。
  • Kibana：一个Web框架，配合Elasticsearch实现存储的日志的可视化，有非常全面的图表分析，有了它，再也不用自己造轮子啦。

　　一个日志系统，就像一个大的工厂。这个工厂里，有人负责收集，有人负责运送，有人负责存储，有人负责排序，有人负责展示汇报。但要让他们乖乖地好好干活，还需要根据每个人的脾气性格调教一番才行。

一个在线Markdown编辑器

发表于 2015-08-20   |  

在线Markdown编辑器StackEdit特点：

  • 开源，免费，搭建在github page上，源码寄存在GitHub。
  • 整合Dropbox和Google Drive，自动同步。
  • 支持一键发布到Google Blog，Tumblr等，甚至通过ssh push到主机上！
  • 可左右或者上下分栏，一边显示Markdown语言一边显示效果。
  • 可以单击左右栏分界线切换至纯写作模式，同样可以收缩顶部工具栏。
  • 在网页顶部工具栏支持加粗、倾斜、超链接、撤消和还原 等等（注意：与Word处理方式有稍许不同）。
  • 不论在左栏还是右栏滚动页面另一侧也会同步。
  • 支持Markdown Extra。
  • 以GIst发布后支持分享（可以在线使用StackEdit阅读）。
  • 多种保存格式，在线缓存（再也不用担心误关浏览器啦）。
  • 详细的说明文档。
  • 界面优美 ; )

　　以上内容转自知乎@苏子雄在知乎用 Markdown 写作用什么文本编辑器？问题下的答案，我做了一点点的修改和补充。

　　这篇blog当然也是用StackEdit来完成的。我只能说：真的好，好出声。有好东西一定要安利给大家。
　　
　　猛戳这儿传送至StackEdit！
